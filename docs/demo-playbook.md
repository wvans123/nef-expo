## 一、测试项 → 演示动作 对照表

> 测试要求原文归纳为 T1/T2/T3 三项。

### T1：能通过服务化接口、类 MCP 接口、意图接口调用 6G 新业务能力（通算、通感等）

| 接口 | 演示动作 | 界面证据 |
|---|---|---|
| 服务化接口（REST） | 「API 直调」选 `target_detection`（通感）或 `compute_offload`（通算）→ 发送 | 左侧请求报文（带 Bearer Key）、右侧 200 响应 + 🔐 鉴权回执 |
| 类 MCP 接口 | 「MCP 接口」点 `tools/list` → 选工具 `tools/call`；或「AF 智能终端」整链路 | JSON-RPC 报文、`subscribed` 标记、调用结果 |
| 意图接口 | 「意图受理」点示例"机器狗巡检，雾天也要看得清" → 发送 → 看状态推进 | 鉴权证据四宫格、Intent ID、阶段流+进度条、执行轨迹回传 |
| （加分）场景级 | 「场景蓝图」选主推场景 → ▶ 一键运行 | 场景级回执：鉴权→受理→转交→编排执行→回执 |

### T2：验证第三方应用访问权限合法性 → 调用内部新业务服务 → 获取结果 → 向第三方反馈

演示串法（一条线讲完）：
1. **权限合法性**：顶栏「注册账号」（注册即签发 Key）→「订阅与鉴权」展示 🔐 四步鉴权流程卡（凭证格式/身份识别/权限范围/审计追踪）；
   - 反例 1：不带 Key 调用 → 401（API 直调把 Key 清掉演示，或终端页未注册状态）；
   - 反例 2：调未订阅能力 → **402 Payment Required**（权限不足≠身份非法，给出订阅/按次两种合法化路径）→ 确认支付 → 通过。
2. **调用内部新业务服务**：意图受理提交 → NEF 转交网络内部 Network Agent → 状态推进（语义解析→业务编排→执行）。
3. **获取结果并反馈第三方**：状态到 completed，执行轨迹+结果摘要回传到 AF 界面（这就是"向第三方反馈服务结果"的呈现位）。

### T3：运营商网络调用第三方能力（网络 Agent 先获取第三方信息，再发起调用，结果反馈网络内部 Agent）

演示动作（「双向开放 · AF」页）：
1. **第三方信息注册**：左侧表单注册能力（名称/类型 ai_model/tool/data_source、描述、**定价**、端点 URL）→ 上架能力超市（third_party 标记）——这就是"网络 Agent 获取第三方的 API/tool 信息"的来源；
2. **内部发现端点**：`POST /internal/mcp`（第二个 MCP Server，面向网络内部 Agent/网元，信任域内免 AF 鉴权）——内部 Agent 经 tools/list 发现第三方工具（含提供方/定价/端点），tools/call 经 NEF 出向网关反向调用并自动入台账；看板「网络内部 Agent 视角」面板有 🔭 一键模拟按钮；
3. **网络发起调用**：右侧看板 ⚡模拟网络调用（顶部链路图逐节点点亮：AF 能力→NEF 超市→Network Agent→NEF 出向网关[鉴权·计费·审计]→AF Endpoint）；
4. **结果反馈网络内部 Agent**：调用日志显示调用方 Agent、耗时、状态、计费；顶部计费汇总卡（总调用/累计计费/70-30 分成/AF 收益）。

| 11-15min | T1-MCP 接口 | tools/list（✓/💳 标记、scenario_*、pipeline_*）→ tools/call；讲标准 /mcp 端点可被 Claude Code 直连 |

## 二点五、关键设计口径（答疑用）

**Q：内部业务是按场景粒度实现的，不是我们的 tool 粒度，怎么办？**
A：tool 粒度是 NEF 的**目录与计费粒度**，不要求业务侧按它实现。对接时按 3.1-B 的场景级触发接口接入即可：一个场景 = 一个 service_id；业务侧返回的 steps 可以只有 1 步（整场景），也可以多步——我们界面按返回的 steps 渲染轨迹，多少步都能演。tool 粒度继续用于 API/MCP 单能力调用的演示（那部分走我们自己的 stub），两边互不阻塞。

**Q：场景需要返回什么结果？NEF 只记录状态够吗？**
A：NEF 定位为**受理/鉴权/计费/状态/审计 + 结果转发**，不复制业务画面。建议每个场景返回三层：
1. `status/progress/steps`（NEF 渲染执行轨迹——已实现）；
2. `result.summary` 一句业务结论 + 3-5 个关键指标（NEF 界面展示——待业务侧给 schema）；
3. `result.detail_url` 业务侧自己演示界面的跳转地址（NEF 只放链接/截图位）。
这样 NEF 界面有"状态+结论+审计"的完整闭环，重内容留给业务侧演示，不抢戏也不缺位。

**Q：外部 AI Agent 怎么直连 NEF 的 MCP？**
```
claude mcp add --transport http nef http://<host>:8000/mcp   --header "Authorization: Bearer <api_key>"
```
已实现 initialize / tools/list / tools/call；未授权工具调用返回计费提醒，携 `_confirm_pay: true` 按次付费执行。

## 三、需要业务同事提供的内部接口（对接需求清单）

> 目前"网络内部执行"为本 Demo 模拟。要接入真实业务（通算/通感新业务），需要业务侧提供以下任一形式的**触发接口**，NEF 侧由我来适配：

### 3.1 业务触发接口（必需，二选一）

**形式 A：同步 HTTP（简单业务，<10s 完成）**
```
POST http://<业务侧host>/v1/services/{service_id}/execute
Body: {"request_id": "...", "params": {...业务参数...}}
返回: {"status": "success|failed", "result": {...}, "latency_ms": 123}
```

**形式 B：异步任务（场景级/长流程业务，推荐）**
```
POST  /v1/tasks            提交：{"request_id", "service_id", "params"} → {"task_id"}
GET   /v1/tasks/{task_id}  查询：{"status": "running|completed|failed", "progress": 0-100,
                                  "steps": [{"name","agent","status","latency_ms"}...],   ← 有这个我就能渲染执行轨迹
                                  "result": {...}}
```
> 我们的 Intent 状态页/场景回执页已经按 B 的数据形状渲染（阶段+steps+result），业务侧照此返回即可零改动接入。

### 3.2 结果内容对接（需要逐业务确认）

每个业务需要业务同事给一份**结果 schema + 1 条样例**，并明确：
- 哪些字段适合在**我们界面**展示（建议：1 行业务结论 + 3-5 个关键指标，如"检测到 3 个目标，置信度 0.93，时延 49ms"）；
- 哪些结果在**业务侧自己的演示界面**呈现（如视频画面、点云）——我们界面只放结论+指标+**跳转/截图位**，不抢业务侧的演示内容；
- 失败/部分成功时给什么文案。

### 3.3 T3（网络调第三方）需要的信息

业务侧（扮演网络内部 Agent 的一方）若要真实回调第三方，需要约定第三方能力的描述格式。我们已有现成格式可直接给他们：
- **API 信息**：`GET /api/v1/capabilities/{tp_id}`（含端点 URL、参数 schema）
- **Tool 信息**：`POST /api/v1/mcp/tools/list`（MCP inputSchema 格式）
- **Skill 信息**：`GET /api/v1/packages/{id}/skill`（编排蓝图 markdown + agent_flow JSON）
- **内部发现/调用（已实现）**：`POST /internal/mcp`（tools/list 发现第三方能力，tools/call 反向调用并登记台账）——内部 Agent 团队可直接对接此接口。

### 3.4 联调资料模板（发给每个业务团队）

| 项目 | 内容 |
|---|---|
| 业务名称 / service_id | 例：机器狗巡检-感知融合 |
| 触发形式 | A 同步 / B 异步 |
| 触发地址与鉴权方式 | URL + token/无 |
| 入参 schema + 样例 | JSON |
| 结果 schema + 样例 | JSON（标注界面展示字段） |
| 预计耗时 | 用于我们的进度条节奏 |
| 失败样例 | 用于异常演示 |

## 四、还可优化的思路（按优先级）

1. ~~真实 MCP Server 直连~~ **已实现**：`POST /mcp`（initialize/tools/list/tools/call），可现场用真实 Claude Code 连 NEF（命令见二点五）。
2. **结果反馈的"业务化卡片"**：按 3.2 拿到各业务结果 schema 后，把 JSON 墙换成业务结论卡（每业务一个渲染模板）。
3. **Intent → 真实任务桥接**：拿到 3.1-B 接口后，把 intent 状态时间线从"定时器模拟"切到真实任务查询，只改 `intent_status()` 一个函数。
4. **审计台账页**：把每次鉴权的 request_id 汇成一张可检索的审计表（测试项"合法性验证"的书面证据，也是运营商关心的合规叙事）。
5. **第三方调用真实回调**：给网络侧一个 `report-call` 回写接口（3.3），把 T3 从"模拟按钮"升级为"业务侧真实发起"。
